1. Introduction

The SZÖRP game is a survival, story, and an adventure game in one. The player will be put in a field and will be required to survive and/or escape the maze surrounding the survival zone. It is set in an imaginary universe with a great story that will be immersive to all those who play it.

In this documentation the reader will get information about the process of making the game.

2. Overview

The game will be available on PC, and Android. And will have different type of game modes to accompany every type of gamer. We chose this idea since not many if any games are made to be like this, so it will be quite a unique experience.

3. Specific Requirements

For the game it is required that the player has Windows 7 (or newer) installed on their computer, or Android 10 (or newer) on their phone and it should be able to work at speeds that random generations won’t throttle the CPU to its limits for performance issues since there will not be lag spikes every ongoing in-game day.

3.1. PC requirements

  1. Windows 7 or newer
  2. A processor with multiple threads, and at least 2 cores
  3. Regarding the GPU, integrated graphics is enough to play the game, however in custom game modes it might not be enough to render the map
  4. ~200 MB of free space and 2 GB of RAM

3.2. Mobile requirements

  1. Android 10 or newer
  2. A 4 core 1.8 GHz processor
  3. At least 900 MHz GPU
  4. ~ 100 MB of free space and 2 GB of RAM

4. Usability

4.1. Learning curve

The game is easy to learn, so it should not take a long time to get to know the game a little better. The game begins with an optional tutorial which explains everything, after that there will always be an option to open a guide that writes down every property in detail.

4.2. Game time

It takes approximately 3 hours to finish a campaign on the default setting, although this time differs between game modes. On a rushed difficulty, and/or an experienced player it might only be 1 hour if they don’t complete every side objective.

4.3. Other preferences

We plan on making a forum for the players where they can put out their ideas for us to consider, the supporters on the other hand have the opportunity to change anything they want in the game.

4.4. GUI

The GUI will be made according to other top/down survival games, so it should be familiar to those who played in this type of genre. Hovering the cursor above elements on the screen will relay information about almost everything on the screen, this includes enemies, animals, crops. Said hover effect can be disabled in the settings.

5. Reliability

5.1. Availability

The application is a single player game so there is no need for a server to run, and anyone can play anytime as long as they have it downloaded on their computer. The game will be available on Steam, and Google Play so the initial download and updates are handled by those services. Since there are no online services involved inside the game there is no maintenance time and limited access.

5.2. Bugs

If at any point in the development, there is a bug which we did not see while making the game we are eager to fix it as soon as possible. We aim to fix any upcoming issues in a 1 – 24 hours range, so the MTTR comes out for about ~12 hours average.

5.3. Updates

We plan on regularly bringing out new updates (this includes smaller updates, like new variety tiles, or even larger ones like new game modes) once a month. We can say that optimally new bugs (errors) can only emerge at these times, so MTBF should be 1 month on average.

6. Performance

6.1. Response times

Some functions in the game operate in a timed environment. We use both a tick system and simple time-based events too. In both scenarios there is a delay since in the game time only passes between frames, so the response time depends on the FPS, but let’s say we play at 60 FPS, the time between frames is 1/60 seconds, so the worst case scenario is that a timed event just ended after a frame got called, so this way the delay is 1/60 seconds or 16.67 milliseconds.

6.2. Multi-threaded operations

While running the game multiple things happen at the same time, these already run parallel thanks to the engine we chose, but we can speed it up even more, by using separate threads to speed up for example the map generation process.

7. Supportability

Creating the game we used well-known programming standards, such as camelCase and PascalCase which are coding norms for better to read code, and tried to avoid recursion where possible, to make it easier to follow the code in real time.

8. Design Constraints

8.1. Hardware limitations

There are some limitations that lessen our ability to create what we want, like for map generation the size, if we could make the tiles smaller while maintaining performance the game should look prettier and more detailed, so we worked hard to make a great middle ground.

8.2. Language limitations

In the game we used C# (With the Godot Engine C# API), which is a fast and reliable language with long term support, but it’s a heavily developed language, so the build times and execution of code snippets are slower than it’s predecessors like C++ which is one of the most regarded programming language for games. Not only this but there are certain things that C# cannot accomplish and relies on external libraries to be able to properly do a task.

8.3. Used software

We used IntelliJ Idea and Visual Studio Code for writing the code, but virtually any IDE can be used for the same purpose if there is an extension that provides the required tools to work with the project.

9. Third-party Components

The game is created in Godot Engine, there are no other external-third-party components implemented yet, only those from the engine itself.
